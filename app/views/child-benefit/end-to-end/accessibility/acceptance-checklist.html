Accessibility acceptance checklist

General accessibility acceptance criteria

All screens that contain content should:

  • Have text with a colour contrast of at least 4.5:1 (Normal-sized text up to 23px/18px bold).
  • Not depend on colour alone to communicate information.
  • Icons with a contrast ratio of 3.0:1 and non-text elements and borders. Images used for decorative purposes are set as background images, preferably (CSS). Images can be marked as with a presentation role when they are not needed for information. See "Using images."
  • Allow users to change their colour preferences (high contrast black and white) without loss of content.
  • Announce any changes to all users. For example, hide and display content for screen readers and sighted users when necessary.
  • When new content appears and is after the current point of focus, consider setting an alert, status or polite role. The instruction in the new content is enough for an AT user to realise they need to navigate forward. A status message when something is removed is to help them rearrange their cognitive map of the screen.

Accessibility acceptance criteria for headings

All headings should:

  • Convey the level so screen reader users can jump through the list of headings to the content they're looking for.
  • Be nested in a way that makes sense to users. See the section on Headings and Labels.
  • Allow users to navigate by headings.
  • Use the caption element to provide a title for the table.

Accessibility acceptance criteria for interactive elements (elements that users can focus on)

Make sure screens are operable with the following:

  • a keyboard (use native HTML form elements)
  • two essentials of a good experience are the ability to discern the location of the keyboard focus easily and to discover where the focus landed after an interaction
  • a mouse
  • screen reader in focus or form mode
  • screen reader shortcuts
  • voice

Some screen readers depend on the user understanding guidance behind specific interactions. For example, Dragon users will have to say actual keyboard commands, e.g. "press the delete key".

To make sure users can operate interactive elements, those elements should:

  • Indicate when content is expanded or collapsed
  • Be large enough to tap with one finger
  • Change appearance when focused. For example, links and form elements have defined focus indicators (yellow and thick black border)
  • Change appearance when users hover their mouse over them (see focus indicators)
  • Not trap focus in a loop from which users cannot exit
  • Not assume that title attribute tooltips are accessible for all users
  • Identify links that open in a new tab or window.

Accessibility acceptance criteria for form elements

All form elements should:

  • Show a message if the form has mandatory fields that must be completed
  • Have a perceivable text label
  • Associate labels and instructions with inputs
  • If appropriate, use the fieldset to group related form fields
  • Show an error message. See Forms and Form errors

Accessibility acceptance criteria for layout

To allow users to make adjustments on their device, screens should work when:

  • Users enlarge the text
  • Users zoom to 400%

Additional information on the acceptance criteria

Colour

(1.4.3 CONTRAST, 1.4.11 NON TEXT CONTRAST, 1.4.1 USE OF COLOR)

  • Check the contrast for all text. Normal-sized text (Up to 23px/18px bold) should have a contrast ratio of 4.5:1
  • Check the contrast for non-text elements, like icons, borders, etc. These should have a contrast ratio of 3.0:1
  • Check your content in Accessibility Display Modes. Activate modes such as Windows High Contrast to check the overall contrast and legibility of page elements like icons, borders, links, etc.
  • Check the contrast for all normal-sized text
  • Check the contrast for all large-sized text
  • Check the contrast for all icons
  • Check the contrast of borders for input elements (text input, radio buttons, checkboxes, etc.)

Layout

(1.4.10 REFLOW)

  • Ensure the content can be viewed without distortion when orientated to a horizontal view or zoomed in for easier reading. Design for responsiveness to accommodate a view or zoomed in for easier reading. Design for responsiveness to accommodate a clean viewpoint at all zoom levels. When a person zooms in to over 300% (using either the mobile operating system or the browser settings), the layout should reflow into one column to allow them to view content easily and not have to scroll sideways to read.
  • There should be no two-dimensional scrolling (vertical and horizontal). There are scenarios when two-dimensional scrolling is okay, for instance, on a table or an image in which meaning is lost without the two dimensions.
  • Ensure that viewport zoom is not disabled

Headings

(2.4.6 HEADINGS OR LABELS)

  • Every page should have at least one heading. Modals do not have to have an H1, but it is recommended they do. Screen readers should announce the text content of elements, such as headings.
  • Use only one H1 element per page or view.
  • Keep headings in a complete and logical order from the top of the page. For example, an h3 element should not appear on a page before the first h2 element. And don't jump headings, skipping from an h1 to an h3 element without an h2 in between.
  • The table's caption should describe the information the table contains.
  • Screen readers should announce HTML elements such as buttons and headings - for example, and they should announce a button as a button, an h2 as a heading level 2, and so on.

Interactive elements and their instructions

  • For interactive components, check if the screen reader gives users instructions on how to interact with them - for example, "To activate, press Enter" if this is the native behaviour of the screen reader for that particular element.
  • Check if users get feedback about any changes to the state of the element - for example if the element expands or collapses

Forms and Form errors

(3.3.1 ERROR IDENTIFICATION)

  • Display input error messages above a form and relate them to the specific fields
  • Use the appropriate colour styling for summary errors
  • All inputs in a form are associated with a corresponding label element
  • Associate input error messaging with the input it corresponds to
  • Make sure that error, warning, and success states are not visually communicated by just colour

Using Images

(1.1.1 NON-TEXT CONTENT)

  • Ensure all eligible images have descriptive text (‘alt’ text) so their meaning in context can be communicated via other forms, such as large print, braille, and speech.
  • For decorative images — you don’t need to supply an alt copy (define ‘null’ instead)

Global

(4.1.2 Landmark Name, Role and Vlaue)

  • Use landmark elements to indicate important content regions if appropriate.
